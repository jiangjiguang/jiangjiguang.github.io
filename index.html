  • 分为普通和物化两种视图

Flink运行时架构

发表于 2021-05-07 更新于 2021-06-20 分类于 flink 阅读次数：

运行时组件

作业管理器（JobManager）

  • 控制一个应用程序执行的主进程
  • jobmanager会先收到应用程序，包括 JobGraph、逻辑数据流图（logical dataflow graph）、所有的类库和jar包等
  • jobmanager会把JobGraph转成ExcutionGraph（物理层面的数据流图），包含了所有可以并发执行的任务
  • jobmanager向ResourceManager请求资源（TaskManager），然后将ExcutionGraph分发到TaskManager上执行
  • 在作业运行过程中，jobmanager负责所有需要中央协调的操作，比如checkpoint的协调

任务管理器（TaskManager）

  • flink中的工作进程，每个TaskManager都有一定数量的slot
  • TaskManager启动之后，会向资源管理器注册它的slot，然后JobManager通过资源管理器可以使用这些slot，来执行task
  • 子task slot共享

资源管理器（ResourceManager）

  • 不是YARN的ResourceManager，而是ApplicationMaster中的ResourceManager
  • 主要负责管理TaskManager中的slot
  • flink为不同的环境和资源管理工具提供了不同的资源管理器

分发器（Dispacher）

  • 可以跨作业运行，它为应用的提交提供了REST接口
  • 当一个作业被提交时，分发器就会启动并将作业移交给一个JobManager
  • 分发器也会启动一个WEB UI，用以展示和监控作业的执行信息
  • 分发器在架构中不是必须的，取决于作业提交的方式

执行图

  • 可以分为4层：StreamGraph -> JobGraph -> ExcecutionGraph -> 物理执行图
  • StreamGraph：根据作业中的Stream API编写的代码生成的最初的图，表示作业的拓扑结构
  • JobGraph：StreamGraph经过chain优化后生成JobGraph，提交给JobManager的数据结构
  • ExcecutionGraph：JobManager将JobGraph并行化生成ExcecutionGraph，ExcecutionGraph是调度层最核心的数据结构
  • 物理执行图：JobManager根据ExcecutionGraph对Job进行调度后，在各个TaskManager部署Task后形成的图，并不是一个具体的数据结构

状态

CheckpointedFunction

  • 状态转换函数的核心接口，该函数持有跨数据元素的状态

  • 该接口提供了针对keyed state 和 operator state 的管理的极大便利

  • 该接口包含两个方法：initializeState snapshotState

  • initializeState

    1、当实现该接口的类被实例化时，调用该方法

  • snapshotState

    1、每当checkpoint 做该函数的状态快照的时候执行

    2、该函数可以作为与外部系统flush/commit/synchronize 的hook

  • kafka consumer使用案例

    initializeState：用来获取状态中的每个 partition的 offset

    snapshotState：获取每个checkpointId的每个 partition的 offset

  • Kafka producer 使用案列

  • TwoPhaseCommitSinkFunction 使用案例

    1、所有实现仅一次语义SinkFunction的建议基类

    2、它通过 CheckpointedFunction 和 CheckpointListener 实现两阶段提交算法

    3、用户应该提供自定义的事务处理

    4、initializeState：

    5、snapshotState：两阶段提交中的预提交

  • hudi使用案例

    initializeState：从instantState中，获取时间轴上最新已经被处理的时间 issuedInstant

    snapshotState：把issuedInstant 保存到 instantState中

CheckpointListener

  • 该接口是被用来与外界进行事务交互，一旦checkpoint完成提交外部事务

  • Checkpoint Subsuming Contract

    1、有两组artifacts

    ready set：下个checkpoint将要被发布的artifacts

    pending set：当本次checkpoint完成时，发布

  • 有两个方法：notifyCheckpointComplete 和 notifyCheckpointAborted

  • notifyCheckpointComplete

    1、通知listener 给定的checkpointId已经完成并且提交

    2、”通知行为”是”尽力而为”的，这意味着可能有时会跳过，因此要遵循”Checkpoint Subsuming Contract”

  • notifyCheckpointAborted

    1、当分布式的checkpoint被终止时，该方法被调用

    2、该方法一般不需要实现

  • TwoPhaseCommitSinkFunction使用案例

Source

SourceFunction

  • 所有source算子的基接口

  • 当source开始发射数据时，run方法是被调用的，run方法可以运行足够长的时间，cancel方法用以结束run方法

  • Timestamps and watermarks

    1、source算子可以给数据指定timestamps，并且可以发射watermarks

    2、这在EventTime时才有效，在其它时间类型下，会忽略时间戳和watermarks

  • 包含两个方法：run 和 cancel

  • run方法

    启动source，并且使用SourceContext 发射数据

  • cancel方法

    一般设置为volatile变量，保证run内部的循环可以结束

SourceContext

  • SourceContext是SourceFunction的内部接口，用于发射数据和watermarks

  • 一共包含六个方法：

    1、collect：发射数据，没有timestamp

    2、collectWithTimestamp

    3、emitWatermark

    4、markAsTemporarilyIdle

    5、getCheckpointLock：应该是类级别锁

    6、close：系统关闭context

RichFunction

  • 所有rich UDF的基类

  • 该类定义了函数的生命周期，并且可以获取运行函数的context

  • 该接口包含5个方法

    1、open：函数的初始化方法，该方法可以使用任务的参数对象

    2、close：用于清理工作

    3、getRuntimeContext：类的运行时信息

    4、getIterationRuntimeContext：获取特定版本的RuntimeContext，

    5、setRuntimeContext：框架用来设置RuntimeContext

算子

StreamOperator

  • 所有的方法保证不被并发调用

  • 该接口共包含9个方法

    2、close：当所有元素被添加到算子后调用，用于flush buffer

    4、prepareSnapshotPreBarrier：当算子应该做snapshot，在发射自己的checkpoint barrier之前调用

    ​ 不做实际的状态持久化，应用于一些小的事务处理

    5、snapshotState：做该算子的状态snapshot

    6、initializeState：

    7、setKeyContextElement1 setKeyContextElement2

    8、getMetricGroup

    9、getOperatorID

SetupableStreamOperator

  • 用来实现访问 context and the output

  • 共包含3个方法：

    2、getChainingStrategy

    3、setChainingStrategy

CheckpointedStreamOperator

Executor内存

  • Driver的内存管理比较简单，和一般的JVM程序区别不大，所以本文重点分析Executor的内存管理
  • 大致分为堆内和堆外内存

堆内内存

  • 通过spark.executor.memory参数设置

  • Executor内运行的并发任务共享JVM堆内内存

  • 堆内内存分为4个部分

    1、Storage内存，主要用于存储Spark的cache数据，例如RDD的cache，Broadcast变量

    ​ 默认为统一内存的0.5

    2、Execution内存，用于存储task执行过程中需要的对象，如Shuffle、Join、Sort、Aggregation等计算过程中的临时数据

    为统一内存 - Storage内存

    3、User内存，用来执行用户的代码

    4、Reserved内存：这部分内存是预留给系统使用，是固定不变的

堆外内存

  • 从概念上分为两个：

    Executor JVM内的堆外内存，主要用于JVM自身，字符串, NIO Buffer等开销

    直接在work节点上的堆外内存，提高Shuffle时排序效率

SparkSQL调优

发表于 2021-04-14 更新于 2021-06-20 分类于 spark 阅读次数：

简介

  • 分布式计算引擎在调优方面有四个主要关注方向，分别是CPU、内存、网络开销和I/O，其具体调优目标如下：

    1、提高CPU利用率

    2、避免OOM

    3、降低网络开销

    4、减少I/O操作

调优参数

executor

  • spark.executor.cores

    作用：单个executor内任务的并发数，这些任务共享executor内存，因此不宜设置过大，容易造成oom

    默认值1，建议值2

  • spark.executor.memory

    作用：各个 executor 使用的堆内内存，不可超过单机的最大可使用内存。

    默认值2g，建议值12g

  • spark.yarn.executor.memoryOverhead

    单位MB，默认值3072

    作用：包括JVM overhead，sort、shuffle以及Netty的堆外内存等。

    限制： spark.executor.memory + spark.yarn.executor.memoryOverhead <= 16G

driver

  • spark.driver.memory

    driver的堆内内存，默认2g，建议12g

  • spark.yarn.driver.memoryOverhead

    driver的堆外内存，默认3072，建议3072

动态分区

  • hive.exec.dynamic.partition

    支持动态分区，默认值false，建议值true

  • hive.exec.dynamic.partition.mode

    当有一个分区是静态分区时：strict 其他设置为：nonstrict

  • hive.exec.max.dynamic.partitions

    允许的分区总数，建议值100000

  • hive.exec.max.dynamic.partitions.pernode

    每个执行节点允许的最大分区数，应该是executor节点

    默认值100，建议值100000

  • hive.exec.max.created.files

    所有map和reduce任务可以产生的hdfs文件的和

    默认值100000，建议值150000

  • hive.error.on.empty.partition

    当有空分区生成时，是否抛出异常，默认值false

join

  • spark.sql.autoBroadcastJoinThreshold

    1、当大表JOIN小表时，如果小表足够小，可以将大表分片，分别用小表和每个大表的分片进行JOIN，最后汇总，能够大大提升作业性能。

    2、如果小表的大小小于该值，则会将小表广播到所有executor中，使JOIN快速完成。如果该值设置太大，则会导致executor内存压力过大，容易出现OOM。

    默认值25M，建议值100M

  • hive.auto.convert.join

    自动将 Reduce 端的 Common Join 转化为 Map Join，从而加快大表关联小表的 Join 速度

orc

  • spark.sql.orc.filterPushdown

    当为true时，开启filter的下推功能

    默认值false，建议值true

  • hive.exec.orc.split.strategy

    ETL：会切分文件,多个stripe组成一个split

    BI：按文件进行切分

    HYBRID：平均文件大小大于hadoop最大split值使用 ETL,否则BI

  • spark.sql.orc.splits.include.file.footer

    官网建议true

  • spark.sql.orc.cache.stripe.details.size

    建议值10000

  • spark.sql.hive.metastorePartitionPruning

    当为true时，一些谓词将被下推到hive的metastore，以便于不匹配的分区能够及早清除。仅仅作用于hive表不能被转换成ilesource relations

  • spark.hadoop.mapreduce.input.fileinputformat.split.maxsize

    读ORC时，设置一个split的最大值，超出后会进行文件切分

    建议值128M

  • spark.hadoop.mapreduce.input.fileinputformat.split.minsize

    读ORC时，设置小文件合并的阈值

    建议值64M

内存管理

  • spark.memory.fraction

    统一内存（storage+execution）所占比例 统一内存=（jvm内存-预留内存 ）* 统一内存所占比例

    保留内存，防止产生oom，默认为300M

    其它内存，用于执行用户的代码的内存 其它内存 = （jvm内存-预留内存 ）- 统一内存

    建议值0.6

  • spark.storage.storageFraction

    storage内存所占比例，用于缓存数据 storage内存= 统一内存 * storage内存所占比例

    execution执行内存，用于缓存在执行shuffle过程中产生的中间数据和直接输出的部分缓存数据

    execution内存=统一内存 - storage内存

    执行内存的空间被对方占用后，可让对方将占用的部分转存到硬盘，然后”归还”借用的空间

    存储内存的空间被对方占用后，无法让对方”归还”

  • 数据缓存方式

    MEMORY_ONLY 仅仅放在内存中，如果内存放不下就要清除

    MEMORY_AND_DISK 优先内存，内存放不下就写到磁盘

    MEMORY_ONLY_SER 以序列化方式存放到内存

    MEMORY_AND_DISK_SER 以序列化方式存放到内存和磁盘

    垃圾回收的代价和堆里的对象数据相关，而不是和数据的字节数相关。

    序列化的方式会把大量对象序列化为一个巨大的缓存区对象，如果缓存的数据量比较大或者有gc时间过长

    可以使用序列化的方式。

  • spark.dynamicAllocation.maxExecutors

  • spark.dynamicAllocation.minExecutors

    每个application可以同时拥有的最小Executor数量，建议值3

shuffle

  • spark.sql.shuffle.partitions

    该参数代表了shuffle read task的并行度

    该参数应该和spark.dynamicAllocation.maxExecutors、spark.executor.cores配合使用应该为上面两个参数的整数倍

  • spark.shuffle.file.buffer

    该参数用于设置shuffle write task的BufferedOutputStream的buffer缓冲大小。将数据写到磁盘文件之前，会先写入buffer缓冲中，待缓冲写满之后，才会溢写到磁盘。

    默认值32k 建议值64k

  • spark.reducer.maxSizeInFlight

    该参数用于设置shuffle read task的buffer缓冲大小，而这个buffer缓冲决定了每次能够拉取多少数据。

    改值太大容易导致reduce oom

    默认值48m 建议值96m

Spark基本知识

发表于 2021-04-14 更新于 2021-06-20 分类于 spark 阅读次数：

基本概念

  • RDD

    只读、分布式、弹性、基于内存

  • 有向无环DAG

    按照宽依赖进行stage划分

  • Application：表示你的应用程序

  • Driver

    1、创建Spark上下文

    2、提交Spark作业（Job）并将Job转化为计算任务（Task）交给Executor计算

    3、协调各个Executor进程间任务调度

  • Executor

    1、在工作节点上执行具体的计算任务（Task），并角计算结果返回给Driver

    2、为需要持久化的RDD提供存储功能

  • Worker：集群中可以运行Application代码的节点。在Standalone模式中指的是通过slave文件配置的worker节点，在Spark on Yarn模式中指的就是NodeManager节点。

  • Task：在Executor进程中执行任务的工作单元，多个Task组成一个Stage

  • Job：包含多个Task组成的并行计算，是由Action行为触发的。也就是每个action会触发一个job

  • Stage：每个Job会被拆分很多组Task，作为一个TaskSet，其名称为Stage

  • DAGScheduler：根据Job构建基于Stage的DAG，并提交Stage给TaskScheduler，其划分Stage的依据是RDD之间的依赖关系

  • TaskScheduler：将TaskSet提交给Worker（集群）运行，每个Executor运行什么Task就是在此处分配的。

  • Application Master

    在YARN中，每个Application实例都有一个ApplicationMaster进程，它是Application启动的第一个容器。它负责和ResourceManager打交道并请求资源，获取资源之后告诉NodeManager为其启动Container。从深层次的含义讲YARN-Cluster和YARN-Client模式的区别其实就是ApplicationMaster进程的区别。
