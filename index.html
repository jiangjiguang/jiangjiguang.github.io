内存探究

发表于 2020-09-08 分类于 linux 阅读次数：

基本概念

主存

也称为物理内存，是计算机的数据存储区域。通常是动态随机访问内存（DRAM）

虚拟内存

  • 是一个抽象概念
  • 大多数虚拟内存是以页的方式实现的
  • 它向每个进程和内核提供巨大的、线性的并且私有的地址空间
  • 它简化了软件开发，把物理内存的分配交给操作系统管理
  • 支持多任务，因为虚拟地址空间被设计成分离的，并且可以超额订购
  • 进程的地址空间由虚拟内存子系统映射到主内存和物理交换设备，内核按需在它们之间移动内存页，称为交换

常驻内存

当前处于主存中的内存

匿名内存

无文件系统位置或路径名的内存，包括进程地址空间的工作数据，称为堆。

地址空间

内存上下文，每个进程和内核都有对应的虚拟地址空间

段

标记为特殊用途的一块内存区域，例如用来存储可执行或者可写的页

页

操作系统和CPU使用的内存单位。它一直以来是4KB或者8KB。允许多种页大小以支持更大的页面尺寸

缺页

无效的内存访问，使用按需虚拟内存时，这是正常事件

换页

  • 换页是将页换入和调出主存，分为称为页面换入和页面换出
  • 由于页的尺寸较小，换页是精确管理和释放主存的手段
  • 有两种类型的换页：文件系统换页和匿名换页

文件系统换页

  • 文件系统换页由读写位于内存中的映射文件页引发，对于使用文件内存映射（mmap()）的应用程序和使用了页缓存的文件系统，这是正常行为。是好的换页。

匿名换页

  • 匿名换页牵涉进程的私有数据：进程堆和栈
  • 被称为匿名是由于它在操作系统中缺乏有名字的地址
  • 匿名页面换出要求迁移数据到物理交换设备或者交换文件，linux称为swapping
  • 匿名换页拖累性能，是坏的换页

按需换页

  • 是指支持按需换页的操作系统将虚拟内存按需映射到物理内存

  • 把CPU创建映射的开销推迟到实际需要或访问时，而不是在初次分配这部分内存时

  • 当访问一个尚未从虚拟映射到物理内存的页时，会发生缺页

  • 如果这个映射可以由内存中其它的页满足，称为轻微缺页

  • 需要访问存储设备的缺页，称为严重缺页

  • 虚拟内存和按需换页的结果是任何虚拟内存也可能处于如下的一个状态：

    A. 未分配

    B. 也分配，未映射

    C. 已分配，已映射到主存（RAM）

    D. 已分配，已映射到物理交换空间（磁盘）

  • 常驻集合大小（RSS）：已分配的主存页（C）大小

  • 虚拟内存大小：所有已分配的区域（B+C+D）

过度提交

  • 是指允许分配超过物理内存和交换设置的总和
  • 依赖于按需换页以及应用程序通常不会使用分配给他们的大部分内存
  • linux可以用可调参数配置过度提交
  • 过度提交的后果取决于内核如何管理内存压力

GC基础知识

引用

Java中提供了4个界别的引用：强引用、软引用、弱引用和虚引用

##强引用（Strong Reference）

  • 在一个线程内，可以直接使用的对象

软引用（Soft Reference）

  • 虚拟机会尽量让软引用的对象存活时间长一些
  • 发生OOM之前会清理所有的软引用对象
  • 虚拟机在GC时会通过算法检查是否回收软引用对象
  • 虚拟机在回收时会将引用存放到一个引用队列

弱引用（Weak Reference）

  • 虚拟机在GC时总是回收弱引用对象
  • 复杂关系的弱引用对象常常需要几次GC才能完成回收

虚引用（Phantom Reference）

  • 虚引用对象get方法返回总是null，因此无法获取被引用的对象

JVM垃圾收集器小结

发表于 2020-09-03 更新于 2020-09-07 分类于 java 阅读次数：

概览

Serial

  • 年轻代收集器
  • 是一个最基本、历史最悠久的单线程收集器
  • 垃圾收集时必须暂停其它工作线程直到收集结束
  • 是client模式下的默认新生代收集器

ParNew

  • 年轻代收集器

  • 是Serial收集器的多线程版本

  • 除了Serial之外，只有它能与CMS收集器配合工作，当使用CMS时，它是新生代默认的收集器

  • JVM配置参数

  • MinorGC发生的条件

Parallel Scavenge

  • 年轻代收集器
  • 使用复制算法、多线程
  • 是一个以吞吐量优先的收集器，
  • 特点是拥有自适应调节策略，设置相应参数给虚拟机设立一个优化目标。

Serial Old

  • 老年代收集器，是Serial的老年代版本
  • 单线程收集器，使用标记-整理算法，主要意义是Client模式下使用
  • 作为CMS收集器的后备预案，在并发收集发生时使用

Parallel Old

  • 老年代收集器
  • Parallel Scavenge收集器的老年代版本，使用多线程和标记-整理算法
  • 在注重吞吐量和CPU资源敏感的场合，优先考虑Parallel Scavenge + Parallel Old的组合

CMS

  • 老年代收集器

  • Concurrent Mark Sweep，并发-标记-清除

  • 以获取最短回收停顿时间为目标

  • 能与之配合使用的年轻代回收器只有Serial和ParNew

  • 是一种基于并发、使用标记清楚算法的垃圾回收器，尽可能让GC线程和用户线程并发执行，减少STW时间

  • 整个过程分为4个步骤

  • JVM配置参数

  • 对象直接进入老年代的4中情况

  • Full GC的触发条件

  • promotion failed && concurrent mode failure

  • CMS的缺点

G1

  • 同时作为新生代和老年代使用

  • 会产生STW，但是通过并发与并行会减少STW的时间

  • 采用标记-整理算法，不会产生内存碎片

  • 新生代和老年代不再是物理隔离的，而是将堆划分为多个大小相等的region，每个region可以当做新生代、也可以当做老年代使用

  • 除了像CMS那样降低停顿时间，还能建立可预测的停顿时间模型

  • 收集过程

垃圾收集器参数

搜集器参数

  • UseSerialGC

    在client模式下的默认值，使用 Serial + Serial Old 收集器

  • UseParNewGC

    使用 ParNew + Serial Old 收集器

  • UseConcMarkSweepGC

    使用 ParNew + CMS + Serial Old 收集器

  • UseParallelGC

    在server模式下的默认值，使用 Parallel Scavenge + Serial Old 收集器

  • UseParallelOldGC

    使用 Parallel Scavenge + Parallel Old 收集器

其它参数

  • -XX:CICompilerCount

    特点代码的编译线程数

  • -XX:+UseCompressedClassPointers

    压缩类指针

  • -XX:+UseCompressedOops

    在64位机器中，压缩普通对象指针，节省空间

  • -XX:+UseFastUnorderedTimeStamps

  • -XX:MinHeapDeltaBytes

    GC导致的堆空间的最小变化（以字节为单位）
