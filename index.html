  • 代码中的优先级最高
  • 代码中资源目录下配置文件的优先级次之
  • 服务中的配置文件的优先级再次之
  • 服务中的默认配置文件的优先级最低

HDFS的写数据流程

写数据流程：

  • Client通过Distributed FileSystem向NameNode请求上传文件，NameNode检查父目录、文件是否已经存在
  • NameNode返回可以上传
  • Client向NameNode请求Block的存储位置
  • NameNode通过计算选择距离近的节点，返回给Client
  • Client通过FSDataOutputStream和相关的DataNode建立通讯管道
  • DataNode应答Client
  • Client以Packet为单位向DataNode1传递Block，DataNode1收到后放到本地内存，然后传给DataNode2、DataNode2传递给DataNode3，DataNode1每传递一个Packet就会放到应答队列等待应答。

DataNode是先写入内存然后在落盘。

节点距离计算

NameNode选择DataNode的因素：

  • 距离近
  • 负载低

节点距离：两个节点到达最近的共同祖先的距离总和。

副本存储节点选择

  • 第一个副本在Client所在的节点上，如果Client在集群外，则随机选一个节点
  • 第二个副本和第一个副本位于相同机架，随机节点
  • 第三个副本位于不同机架，随机节点

HDFS的读数据流程

HDFS架构简介

发表于 2020-06-23 分类于 hadoop 阅读次数：

HDFS的优点和缺点

优点

  • 高容错性

    多副本

    副本丢失，可以自动恢复

  • 适合处理大数据

    能够处理PB级别的数据

    能够处理百万以上的文件数量

  • 可构建的廉价的机器上

缺点

  • 不适合低延迟数据访问
  • 无法高效地对大量小文件进行存储
  • 不支持并发写入，文件随机修改

HDFS的架构

NameNode

  • 管理HDFS的名称空间
  • 配置副本策略
  • 管理数据块（Block）映射信息
  • 处理客户端读写请求

DataNode

NameNode下达命令，DataNode负责执行实际的操作

  • 存储实际的数据块
  • 执行数据块的读写操作

Client客户端

  • 文件切分 文件上传到HDFS的时候，Client将文件切分成一个个的Block，然后进行上传
  • 与NameNode交互，获取文件的位置信息
  • 与DataNode交互，读取或者写入数据
  • 通过命令管理HDFS，比如NameNode格式化等
  • 通过命令访问HDFS，比如对HDFS进行增删查改等操作

Secondary NameNode

不是NameNode的热备，当NameNode挂掉时并不能替换NameNode

  • 辅助NameNode，分担其工作量，比如定期合并fsimage和edits文件，并推送给NameNode

  • 辅助恢复NameNode

HDFS小文件

发表于 2020-06-23 分类于 hadoop 阅读次数：

简介

HDFS的缺点之一就是无法高效地对大量小文件进行存储：

  • 每个小文件会占用150个byte，会占用大量的NameNode的内存存储文件的目录和块信息。
  • 小文件存储的寻址时间会超过读取时间

HDFS文件块大小

HDFS中的文件在物理上是分块（Block）存储的，块大小通过dfs.blocksize参数设置的，生产中设置的是268435456，即256MB。

寻址时间为传输时间的1%时，为最佳状态。

影响块大小的最主要因素是磁盘传输速率：

  • 块大小=1s * 磁盘传输速率
  • 目前磁盘普遍传输速率为100MB/s，若磁盘传输速率高，应该把块大小提高。

HDFS的块应该合理设置：

  • 设置太小，会增加寻址时间，程序查找块开始位置比较耗时
  • 设置太大，从磁盘传输数据的时间会明显大于定位块开始位置所需的时间。导致程序在处理这块时间时，非常慢

如果副本数设置为10，那么每个节点最多存储一个副本，3个节点就存储3份，每增加一个节点就会增加一个副本

export和source

发表于 2020-01-14 更新于 2020-06-22 分类于 linux 阅读次数：

export作用：设置或显示环境变量。

语法：export [-fnp][变量名称]=[变量设置值]

参数：

  -f 代表[变量名称]中为函数名称。

  -n 删除指定的变量。变量实际上并未删除，只是不会输出到后续指令的执行环境中。

  -p 列出所有的shell赋予程序的环境变量。

  用户登录到linux终端后，系统将启动一个用户shell。在这个shell中，可以使用或者声明变量，也可以创建并运行 shell脚本程序。运行shell脚本程序时，系统将创建一个子shell。子shell可以使用父shell的所有环境变量。

  在子 shell中定义的变量只在该子shell内有效。如果在一个shell脚本程序中定义了一个变量，当该脚本程序运行时，这个定义的变量只是该脚本程序内的一个局部变量，其他的shell不能引用它，要使某个变量的值可以在其他shell中被使用，可以使用export命令对已定义的变量进行输出。

  简单来讲，export的作用就是设置环境变量，从而可以在本shell或者子shell中使用该变量，但是不能在父shell中使用。

  如果使用source执行子脚本设置环境变量，那么该环境变量也可以在其父环境（当前环境）下使用

结论：

  1.bash命令执行脚本时是在一个子shell环境运行的，脚本执行完后该子shell自动退出

  2.export定义的环境变量才会被复制到子 shell中

  3.不用export定义的变量只对该shell有效，对子shell也是无效的

  4.使用source执行脚本是在当前环境运行的，所以脚本中的环境变量也对当前shell有效
