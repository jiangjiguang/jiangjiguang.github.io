flink为什么不能使用prometheus pushgateway

发表于 2020-07-20 分类于 flink 阅读次数：

简介

​ flink可以使用多种Reporter向外部系统暴露指标（metrics），比如JMX、InfluxDB、prometheus等。由于公司运维的监控告警系统采用的是prometheus监控组件，而且公司对这方面技术存储比较多，所以实时计算平台的指标采集当仁不让的采用prometheus监控组件。

​ 但是即使采用prometheus收集任务指标，又有两种方式可供选择，即exporter 和 pushgateway。

  • flink-conf.yaml中pushgateway配置
  • flink-conf.yaml中exporter配置方式

prometheus pushgateway指标收集

​ 在开始搭建实时计算平台时，根据运维的建议和业界的调研最终选择了pushgateway的指标收集方式，刚开始时指标收集和报警都比较良好。但是随着任务数量的增多，任务数达到600的时候，prometheus每次pushgateway采集数据的时间达到70~80秒，而且还伴发着偶发性的超时（超时时间90秒）报警延时大概5分钟。为什么会这样呢？prometheus官网不建议pushgateway作为大量的指标收集方式，实时计算平台当任务数达到600时，其指标数量已经达到22000个。这时pushgateway 进程已经占用了45G内存、CPU负载达到300%。当从pushgateway把所有的指标下载的到本地磁盘后，只占3G左右的磁盘空间，为什么在占用内存会比磁盘大了一个数量级呢？那么随着指标数量的增加，pushgateway的内存是不是线性增加的呢？通过模拟线上指标数据，对pushgateway进行了测试，得出一个结论：pushgateway所占内存会随着指标的数量增加而线性增加。

解决方案：

当时想到3个解决方案，但是都被一一排除：

  • 方案一

    精简哪些指标、怎么精简指标、精简之后会造成哪些影响，这些都可能会耗费大量的精力，而且效果不一定明显

  • 方案二

    如何pushgateway使用多台，那么怎么把不同指标达到不同的pushgateway，随着任务数量的增多pushgateway机器不够了还需要继续扩容？当下600个任务已经占用了45G内存，采集时间太长，而且偶发性超时怎么解决

  • 方案三

    如果使用InfluxDB代替，那么开发人力成本、和现有系统的结合，又将是面临的问题

pushgateway

指标一致性检查

然而如果大量的指标是被频繁推送的pushgateway，pushgateway聚合的成本非常高，那么push的时间将会非常长。可以使用–push.disable-consistency-check 来节省一致性检查的成本，但是这将导致不一致的指标。设置了该参数，在抓取时任然会进行一致性检查，但是只要有不一致的指标，那么所有的抓取都会失败。 https://github.com/prometheus/pushgateway/issues/313

时间戳

​ 如果指标被推送到pushgateway的时间是t1，你可能错误地认为prometheus在同样的时间抓取他们，事实上，prometheus赋给指标的时间戳是抓取的时间。在prometheus的视角中，prometheus可以在任何时间抓取指标，直到指标被删除。验证：prometheus有时是宽容的，但是如果在5分钟内得不到一个指标的任何采样，那么将其视为已删除。pushed group（推送组）：按照推送组聚合，其实就是指标名称，每个推送组都有push_time_seconds和push_failure_time_seconds。

  • pushgateway不需要任何配置，开箱即用
  • 默认情况下，所有指标均存储在内存中，即如果重新，那么将丢失所有指标。可以通过–persistence.file 持久化于磁盘路径

系统指标

系统方法

  • PUT

    用来推送一组指标，在URL路径中的所有grouping key是被PUT的指标替换？啥意思？没看懂，不是说grouping key 是URL中的指标吗 怎么又被body中的替换呢如果body是空，那么意味着删除所有grouping key的指标，与DELETE不同的是，PUT会更新push_time_seconds

  • POST

    和put方法基本相同，区别是：
    1.仅仅是新推送的并且名称一样才能被替换？查一下具体区别？
    2.如果body是空，push_time_seconds也会更新，但是不会更新任何先前推送的指标值，也就是不会删除。

  • DELETE

    请求不能包含body，所有匹配URL中grouping key的指标将被删除。成功码总是202，和PUT/POST相比具有最高执行优先级。

当任务下线时，flink只会删除JobManager的指标，不会删除TaskManager的指标。
