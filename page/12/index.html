Java反射实践

发表于 2020-08-12 更新于 2021-06-20 分类于 java 阅读次数：

简介

  • 在程序的运行过程中，可以动态创建对象

  • 反射的基石：字节码文件对象，Class类的对象

    1、字节码文件：源文件-> 编译 -> 字节码文件（.class文件）

    2、字节码文件对象：jvm把字节码文件加载到jvm内存中后，jvm就认为其实一个字节码文件对象

  • 获得字节码文件对象

    1、Object类的getClass方法

    2、类型.class属性

    3、Class.forName(“类的路径”)

    4、同一个类的字节码文件只会被加载一次，因此上述三种方式得到的class类是同一个

  • 反射的优点

    1、java可以动态的加载未知的外部配置对象，临时生成字节码进行加载使用，使代码更灵活，极大地提高应用的扩展性。

  • 反射的缺点

    1、性能问题，会慢50~100倍

    2、使用反射是代码逻辑不清晰

    3、安全问题，暴露了私有变量

泛型擦除

  • Java泛型是在编译器层次上实现的，在编译生成字节码的期间，所有的泛型信息都会被擦除掉

  • 在JVM生成的字节码中是没有泛型信息的

  • 那么泛型被擦除后，保留在字节码中的是什么类型呢？

    1.原始类型 类型被擦除后，保留在字节码中的类型变量的真正类型。

    2.无论何时定义一个泛型，相应的原始类型会被自动提供。

    3.如果无限定类型变量，其原始类型就是Object

    4.如果有限定类型变量，那么其原始类型就是第一个边界的类型变量

    5.在不指定泛型的情况下，泛型变量的类型为该方法中的几种类型的同一父类的最小级，直到Object

    6.在指定泛型的情况下，该方法的几种类型必须是该泛型的实例的类型或者其子类

  • Java编译器是通过先检查代码中泛型的类型，然后在进行类型擦除，再进行编译

  • 类型检查是针对引用的，不是针对真正的实例对象

  • 虚拟机巧妙的使用了桥方法，来解决了类型擦除和多态的冲突

基本概念

  • 信号(signal)是一种软中断，是进程间异步通讯的一种方式。

    每个信号都有一个编号和一个宏定义名称，查看方式：cat /usr/include/bits/signum.h

  • Linux系统共定义了64种信号，分为可靠信号和不可靠信号

    不可靠信号：称为非实时信号、不支持排队，信号可能丢失，比如发送多次相同的信号, 进程只能收到一 次。取值范围：1~31

    可靠信号：称为实时信号、支持排队，信号不会丢失，发多少次, 就可以收到多少次。取值范围：32~64

    Stop表示停止当前进程

信号产生

信号的来源可分为：硬件类和软件类

硬件方式

  • 用户输入

    ctrl-c 发送 SIGINT 中断信号给前台进程组中的所有进程。常用于终止正在运行的程序。
    ctrl-z 发送 SIGTSTP 进程停止信号给前台进程组中的所有进程，常用于挂起一个进程。
    ctrl-d 不是发送信号，而是表示一个特殊的二进制值，表示 EOF。
    ctrl-\ 发送 SIGQUIT 退出信号给前台进程组中的所有进程，终止前台进程并生成 core 文件。

  • 硬件异常

    由硬件检测到并通知内核、然后内核向当前进程发送适当的信号，例如：

    1、当前进程执行了除以0的指令，CPU的运算单元会产生异常，内核将这个异常解释为SIGFPE浮点异常信号发送给进程

    2、当前进程访问了非法内存地址，MMU会产生异常，内核将这个异常解释为SIGSEGV无效内存访问信号发送给进程。

软件方式

  • 进程间

    一个进程调用kill(2)函数可以发送信号给另一个进程。

    可以用kill(1)命令发送信号给某个进程，kill(1)命令也是调用kill(2)函数实现的，如果不明确指定信号则发送SIGTERM信号，该信号的默认处理动作是终止进程。

  • 编程语言

    kernel,使用 kill_proc_info(）等

    native,使用 kill() 或者raise() 等

    java,使用 Procees.sendSignal()等

信号注册和注销

信号注册

​ 在进程task_struct结构体中有一个未决信号的成员变量 struct sigpending pending。每个信号在进程中注册都会把信号值加入到进程的未决信号集。

  • 非实时信号发送给进程时，如果该信息已经在进程中注册过，不会再次注册，故信号会丢失
  • 实时信号发送给进程时，不管该信号是否在进程中注册过，都会再次注册。故信号不会丢失

信号注销

  • 非实时信号：不可重复注册，最多只有一个sigqueue结构；当该结构被释放后，把该信号从进程未决信号集中删除，则信号注销完毕
  • 实时信号：可重复注册，可能存在多个sigqueue结构；当该信号的所有sigqueue处理完毕后，把该信号从进程未决信号集中删除，则信号注销完毕

信号处理

​ 内核处理进程收到的signal是在当前进程的上下文，故进程必须是Running状态。

​ 当进程唤醒或者调度后获取CPU，则会从内核态转到用户态时检测是否有signal等待处理，处理完，进程会把相应的未决信号从链表中去掉。

​ 信号处理动作称为信号递达（Delivery）。信号从产生到递达之间的状态，称为信号未决（Pending）

  • signal信号处理时机

    内核态 -> signal信号处理 -> 用户态

    在内核态，signal信号不起作用

    在用户态，signal所有未被屏蔽的信号都处理完毕

    当屏蔽信号，取消屏蔽时，会在下一次内核转用户态的过程中执行

  • 处理方式

    默认：接收到信号后按默认的行为处理该信号，这是多数应用采取的处理方式

    自定义：用自定义的信号处理函数来执行特定的动作

    忽略：接收到信号后不做任何反应

  • 信号安装

    进程处理某个信号前，需要先在进程中安装此信号。安装过程主要是建立信号值和进程对相应信息值的动作。

    信号安装函数:

    1、signal()：不支持信号传递信息，主要用于非实时信号安装

    2、sigaction()：支持信号传递信息，可用于所有信号安装

    函数原型：

    int sigaction(int signum, const struct sigaction *act, struct sigaction *oldact)

    1、signum：要操作的signal信号

    2、act：设置对signal信号的新处理方式

    3、oldact：原来对信号的处理方式

    4、返回值：0 表示成功，-1 表示有错误发生

    其中 sigaction结构体：

    1、sa_handler：信号处理函数

    2、sa_mask：指定信号处理程序执行过程中需要阻塞的信号

    3、sa_flags：标示位

    • SA_RESTART：使被信号打断的syscall重新发起。
    • SA_NOCLDSTOP：使父进程在它的子进程暂停或继续运行时不会收到 SIGCHLD 信号。
    • SA_NOCLDWAIT：使父进程在它的子进程退出时不会收到SIGCHLD信号，这时子进程如果退出也不会成为僵 尸进程。
    • SA_NODEFER：使对信号的屏蔽无效，即在信号处理函数执行期间仍能发出这个信号。
    • SA_RESETHAND：信号处理之后重新设置为默认的处理方式。
    • SA_SIGINFO：使用sa_sigaction成员而不是sa_handler作为信号处理函数

  • 信号发送

    1、kill()：用于向进程或进程组发送信号

    2、sigqueue()：只能向一个进程发送信号，不能像进程组发送信号；主要针对实时信号提出，与sigaction()组合使用，当然也支持非实时信号的发送

    3、alarm()：用于调用进程指定时间后发出SIGALARM信号

    4、setitimer()：设置定时器，计时达到后给进程发送SIGALRM信号，功能比alarm更强大

    5、abort()：向进程发送SIGABORT信号，默认进程会异常退出

    6、raise()：用于向进程自身发送信号

  • 信号函数

    信号集操作函数：

    1、sigemptyset(sigset_t *set)：信号集全部清0

    2、sigfillset(sigset_t *set)： 信号集全部置1，则信号集包含linux支持的64种信号

    3、sigaddset(sigset_t *set, int signum)：向信号集中加入signum信号

    4、sigdelset(sigset_t *set, int signum)：向信号集中删除signum信号

    5、sigismember(const sigset_t *set, int signum)：判定信号signum是否存在信号集中

    信号阻塞函数：

    1、sigprocmask(int how, const sigset_t *set, sigset_t *oldset))； 不同how参数，实现不同功能

    ​ SIG_BLOCK：将set指向信号集中的信号，添加到进程阻塞信号集

    ​ SIG_UNBLOCK：将set指向信号集中的信号，从进程阻塞信号集删除

    ​ SIG_SETMASK：将set指向信号集中的信号，设置成进程阻塞信号集

    2、sigpending(sigset_t *set))：获取已发送到进程，却被阻塞的所有信号

    3、sigsuspend(const sigset_t *mask))：用mask代替进程的原有掩码，并暂停进程执行，直到收到信号再恢复原有掩码并继续执行进程

类生命周期

发表于 2020-08-05 更新于 2021-06-20 分类于 java 阅读次数：

生命周期概述

​ 按照JVM规范，一个Java文件从被加载到被卸载共分为5个阶段：

  • 加载

    1.加载就是将Java类的字节码文件加载到机器内存中，并在内存中构建出Java类的原型-类模板对象。

    2.类模板对象就是Java类在JVM内存中的一个快照，JVM将从字节码文件中解析出常量池、类字段、类方法等信息存储到类模板中

    3.类模板在JDK被存储在metaSpace区

    4.JVM会对字节码文件格式进行验证

  • 链接（验证+准备+解析）

    1.链接的主要作用是将字节码指令中对常量池中索引引用转换为直接引用。

    2.验证是指由字节码信息触发进行反向验证：

    验证字节码文件中的类名是否能够找到对应的类模板

  • 初始化

    并非是对类进行实例化，而是执行类的<client>方法，该方法仅能由Java编辑器生成并由JVM调用。

    当Java类中包含由static修饰的静态字段或者代码块时，编译后的便会在字节码文件中包含一个<client>方法

  • 使用

  • 卸载

类加载

1.类加载的目标就是在JVM方法区创建一个与Java类结构对等的instanceKcass的实例对象。

2.mirror镜像类也是一个instanceKcass的实例对象

3.java.lang.Class是为了被程序调用，instanceKcass是为了被JVM内部访问

4.反射工具类都是基于java.lang.Class这个内部镜像实现的

5.类加载的主要流程：

​ a.读取魔术与版本号

​ b.解析常量池

​ c.解析字段信息

​ d.解析方法

​ e.创建与Java类对等的内部对象instanceKlass

​ f.创建Java镜像类，create_mirror()

Java主类加载机制

1.包含main()主函数的类就是Java主类，由JVM自动触发加载。

2.JVM执行完自身的若干初始化逻辑之后，第一个加载的便是Java主类。

3.Java主类的类加载器模式是系统加载器。

类加载器的加载机制

1.通过继承java.lang.ClassLoader可以自定义类加载器，自定义类加载器的loadClass()方法仅仅完成了声明周期的加载阶段

2.通过反射java.lang.Class.forName(String)完成的类加载，包含了声明周期的加载、验证、准备、解析和初始化阶段

3.import仅仅是个语法糖，没有关联任何运行时行为，更不会导致类加载

4.使用new指令加载类时，如果一个类尚未被加载或者链接，那么会进入慢分配流程

类加载器

1.Java体系中定义了3种类加载器：

​ Bootstrap ClassLoader(引导类加载器，BCL)：加载指定的JDK核心类库，无法由Java应用程序直接引用。负责加载：

​ a.%JAVA_HOME%/jre/lib目录

​ b.-Xbootclasspath参数指定的目录

​ c.系统属性sun.boot.class.path指定的目录中的特定名称的jar包

​ Extension ClassLoader(扩展类加载器，ECL)：加载扩展类，扩展JVM的类库。负责加载：

​ a.%JAVA_HOME%/jre/lib/ext目录

​ b.系统属性java.ext.dirs所指定的目录中的所有类库

​ System ClassLoader(系统类加载器，SCL)：加载Java应用程序类库，加载类库的路径由系统环境变量ClassPath、-cp或系统属性java.class.path指定

引导类加载器

1.使用C++实现，所有字段和函数都使用static修饰，因此该加载器不需要实例化

2.Java类编写的加载器无法继承引导类加载器

3.没有父加载器

扩展类加载器

1.所有Java体系的类加载器都是继承自java.lang.ClassLoader

2.类名是ExtClassLoader

3.扩展类加载器的父加载器是null，并不是引导类加载器

系统类加载器

1.系统类加载器的父加载器是扩展类加载器

双亲委派

1.引导类加载器、扩展类加载器和系统类加载器之间没有直接的继承关系

2.Java体系的扩展类加载器、系统类加载器和用户自定义类加载器都继承自java.lang.ClassLoader

3.委托加载关系是通过java.lang.ClassLoader.parent实现

4.双亲委派机制在java.lang.ClassLoader.loadClass(String,boolean)接口中实现：

​ a.先在当前加载器的缓存中查找有无目标类，如果有，直接返回

​ b.判断当前加载器的父加载器是否为空，如果不为空，则调用parent.loadClass(name,false)接口

​ c.如果当前加载器的父类加载器为空，则调用findBoostraopClassOrNull(name)接口，让引导类加载器进行加载

​ d.如果上述未加载成功，则调用findClass(name)接口进行加载

5.双亲委派机制从本质上规定了类加载的顺序：

​ a.引导类加载器先加载

​ b.若加载不到由扩展类加载

​ c.若还是加载不到，才会由系统类或自定义类加载器加载

6.如果自定义类加载器重写loadClass方法，会不会破坏双亲委派机制？答案是不会的

​ 会判断所要加载的目标类全限定名是否以java开始，如果是，则抛出异常

​ 因此所有的自定义类加载器只能重写java.lang.ClassLoader.findClass接口

类初始化

1.并非是对类进行实例化，而是执行类的<client>方法，该方法仅能由Java编辑器生成并由JVM调用。

2.当Java类中包含由static修饰的静态字段或者代码块时，编译后的便会在字节码文件中包含一个<client>方法。

3.每个类都有一个类加载器，并且不同的加载器加载的同一个类无法相互转换。

类使用

类实例分配

1.实例化类的方式：java.lang.Class.newInstance()、new

​ 在new的实现过程中，对内存分配、线程并发控制、垃圾回收等优化。

2.类实例化过程：

​ a.若Java类尚未被解析，则直接进入慢分配，不会使用快速分配策略

​ b.快速分配，如果没有开启栈分配或不符合条件则会进行TLAB分配

​ c.快速分配，如果TLAB分配失败，则尝试eden区分配

​ d.如果eden区分配失败，则会进入慢分配流程

​ e.如果对象满足了直接进入老年代的条件，那就直接分配在老年代

​ f.快速分配，对于热点代码，如果开启逃逸分析，JVM则会执行栈上分配或标量替换等优化方案

逃逸分析

  • 通过对象作用范围的分析，为其他优化手段提供分析数据

  • 逃逸分析包括：

    1、全局变量赋值逃逸

    2、方法返回值逃逸

    3、实例引用逃逸

    4、线程逃逸：赋值给类变量或可以在其它线程中访问的实例变量

  • 开启逃逸分析

    -XX:+DoEscapeAnalysis

  • 总结

    逃逸是指在方法内部被创建的对象不仅在方法内部而且还在方法外部被其它变量引用。坏处是，方法执行完毕之后，在方法内创建的对象无法被GC回收。

    ​ 如果确定不会逃逸，那么就能对方法内创建的对象进行多种优化，包含内存分配和线程同步

    ​ 逃逸分析是在运行时进行的，原因是动态生产字节码、调用本地函数、反射、方法拦截等致使不能在编译期间的静态化设施。

    ​ 逃逸分析是很耗费内存和CPU资源的，因此JVM不能对所有方法的变量进行逃逸分析，只有JVM触发即时编译（JIT）时才会进行逃逸分析。

    ​ 即时编译（JIT）是一种在运行时将字节码翻译为机器码，从而改善字节码编译语言性能的技术，即时编译基于逃逸分析的优化策略主要是标量替换和栈上分配，都不会在堆上分配实例化对象。

    ​ 可以通过jmap -histo命令观察开启/关闭逃逸分析的实例总数

标量替换

  • 标量是指不可分割的量，基本数据类型和引用都是标量

  • 不会发生逃逸，且不是标量时，JVM不会创建该对象，而会将该对象成员变量分解若干个被这个方法使用的成员变量所代替。这些代替的成员变量在栈帧或寄存器上分配空间。

  • 开启标量替换

    -XX:+EliminateAllocations：开启标量替换

    -XX:+PrintEliminateAllocations：查看标量替换情况

栈上分配

  • 如果一个类没有发生逃逸，则实例对象直接分配在方法栈上。在栈上分配的对象实例，会随着方法执行结束，方法栈空间的被回收而回收，从而不需要GC来回收
  • 栈空间是有限的，因此不能分配字段多的大Java对象
  • 栈上分配依赖于逃逸分析和标量替换

同步消除

  • 如果对象没有出现线程逃逸，那么对象的读写就不会存在资源的竞争，那么就可以清除对象的同步锁

  • 开启同步消除

    -XX:+EliminateLocks

TLAB(Thead Local Allocation Buffer)

  • 即线程本地分配缓存区，是一个线程专用的内存分配区域，线程私有的，位于eden区。

  • 各个线程在各自的TLAB申请空间，无需加锁，是一中“空间换时间”的策略。

  • 堆是全局的，每个线程都可以在堆上申请空间，而后必须在线程间进行同步，当多线程在堆上申请内存时，势必出现竞争、同步、阻塞等。

  • 开启TLAB

    -XX:-UseTLAB

Java对象分配

  • 分配优先级

    栈上分配 -> TLAB分配 -> 直接进入老年代 -> eden区分配
