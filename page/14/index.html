flink为什么不能使用prometheus pushgateway

发表于 2020-07-20 更新于 2021-06-20 分类于 flink 阅读次数：

简介

​ flink可以使用多种Reporter向外部系统暴露指标（metrics），比如JMX、InfluxDB、prometheus等。由于公司运维的监控告警系统采用的是prometheus监控组件，而且公司对这方面技术存储比较多，所以实时计算平台的指标采集当仁不让的采用prometheus监控组件。

​ 但是即使采用prometheus收集任务指标，又有两种方式可供选择，即exporter 和 pushgateway。

  • flink-conf.yaml中pushgateway配置
  • flink-conf.yaml中exporter配置方式

prometheus pushgateway问题

​ 在开始搭建实时计算平台时，根据运维的建议和业界的调研最终选择了pushgateway的指标收集方式，刚开始时指标收集和报警都比较良好。但是随着任务数量的增多，任务数达到600的时候，prometheus每次pushgateway采集数据的时间达到70~80秒，而且还伴发着偶发性的超时（超时时间90秒）报警延时大概5分钟。为什么会这样呢？prometheus官网不建议pushgateway作为大量的指标收集方式，实时计算平台当任务数达到600时，其指标数量已经达到22000个。这时pushgateway 进程已经占用了45G内存、CPU负载达到300%。当从pushgateway把所有的指标下载的到本地磁盘后，只占3G左右的磁盘空间，为什么在占用内存会比磁盘大了一个数量级呢？那么随着指标数量的增加，pushgateway的内存是不是线性增加的呢？通过模拟线上指标数据，对pushgateway进行了测试，得出一个结论：pushgateway所占内存会随着指标的数量增加而线性增加。

从而得出以下结论：

解决方案：

当时想到3个解决方案，但是都被一一排除：

  • 方案一

    精简哪些指标、怎么精简指标、精简之后会造成哪些影响，这些都可能会耗费大量的精力，而且效果不一定明显

  • 方案二

    如何pushgateway使用多台，那么怎么把不同指标达到不同的pushgateway，随着任务数量的增多pushgateway机器不够了还需要继续扩容？当下600个任务已经占用了45G内存，采集时间太长，而且偶发性超时怎么解决

  • 方案三

    如果使用InfluxDB代替，那么开发人力成本、和现有系统的结合，又将是面临的问题

值得一提的是：当任务下线时，flink只会删除JobManager的指标，不会删除TaskManager的指标。

总结起来，当前实时计算平台面临两个问题

  • 指标如何采集，即寻找pushgateway的替代方案
  • 报警延时比较高，达到5~6分钟

基本概念

记录（Record）

  • 数据库加载和存储的最小单位
  • 一条记录可以使 a Document, a Blob, a Vertex or even an Edge

###文档（Document）

  • 文档是最灵活的记录（Record）类型
  • 文档既可以是有模式的，也可以使无模式的

###二进制（BLOB）

  • BLOB是二进制形式的记录（Record）类型
  • v2.2版本之前也叫RecordBytes

顶点（Vertex）

  • 图数据库中最基本的单元是node，在OrientDB中称为vertex
  • vertex也是文档（Document），所以可以存储embedded records 和 arbitrary properties

边（Edge）

  • 图数据库中，arc用于连接nodes，在OrientDB中称为edge
  • edge是双向的，仅仅能连接两个vertex
  • edge又分为两类：regular 和 lightweight
  • regular Edge 是个文档（Document）
  • Lightweight Edge 不是文档

Record ID

  • 每当OrientDB产生一个record，同时自动生成一个唯一的标识符，称为Record ID
  • Record ID由集群（cluster）和位置（position）组成：#<cluster>:<position>
  • Cluster Identifier 指明Record属于哪个Cluster，正数表示持久化存储，负数表示临时存储
  • position 指明Record在Cluster中的绝对位置

Record Version

  • 记录（Record）有自己的版本号，每当更新后会递增

##Class

  • Class用于定于Record
  • Classes can be schema-less, schema-full or a mix
  • 和java中一样，类是可继承的，可以继承父类的属性
  • 每个非抽象类必须有至少一个自己的clusters (data files)，也可以有多个clusters
  • 当对类执行查询时，自动传播到该类的所有clusters
  • 创建Record时，该Record选择保存的clustersconfigurable strategy
  • 当新建Class后，默认创建一个同名的持久化的cluster

Abstract Class

  • 和java中的抽象类的概念类似
  • 不能实例化

Cluster

  • 类Class提供了组织数据的逻辑结构
  • 集群clusters提供了存储数据的物理结构或内存空间，类似于关系型数据库中的表概念
  • 当创建Class时，也会创建物理的clusters，用于存储该class的数据
  • cluster的命名方式是类Class转小写

Materialized View

  • 是a persistent object that contains the result of a query
  • 在SQL查询中，可以作为class的等价
  • 可以使配置为只读和可更新的
  • 向class一样，可以有索引

#Relationships

  • 支持两种类型的relationships，referenced 和 embedded

###Referenced Relationships

  • 关系型数据库中，table通用join连接，这可能会在计算资源上造成高昂的代价
  • OrientDB通过直接存储目标对象的连接
  • 使用LINK类型表示引用连接关系：LINKLIST、LINKSET、LINKMAP
  • Edges仅仅连接两个vertices
  • 两个vertice都有Record ID

Embedded Relationships

  • 和Referenced Relationships不同，被嵌入的record 没有自己的Record ID
  • 当嵌入的Record删除后，被嵌入的Record也会被删除
  • EMBEDDEDLIST、EMBEDDEDSET、EMBEDDEDMAP表示连接关系

Inverse Relationships

  • OrientDB中所有的Edges都是双向的

图数据库概览

发表于 2020-07-13 更新于 2021-06-20 分类于 图数据库 阅读次数：

图是什么

  • 图将实体表现为节点，实体与其它实体连接的方式表现为联系
  • 天生不需要模式

现在最流行的图模型形式-带标签的属性图

  • 它包含节点和联系
  • 节点上有属性（键值对）
  • 节点可以有一个或多个标签
  • 联系有名字和方向，并总是有一个开始节点和一个结束节点
  • 联系也可以有属性
  • 使用labels表示节点在图中扮演的角色

图领域的分类

  • 主要用于联机事务图的持久化技术，通常直接实时地被应用程序访问
  • 主要用于离线图分析的技术，通常按一系列步骤执行

图数据库

  • 一般用于事务（OLTP）系统中
  • 原生图存储的好处是，它的栈是专门为性能和扩展性设计建造的
  • 非原生图存储通常建立在非常成熟的非图后端之上
  • 原生图处理（免索引邻接）虽然在遍历查询时性能优势很大，但代价是一些未使用遍历的查询会比较难，并且需要占据巨大的内存
  • 联系是图数据模型中的一等公民
  • 图数据库通过将节点和联系的简单抽象组装为相互关系的结构，是我们能够建造任意复杂的模型，形象地映射我们的问题域
  • 图天生是可扩展的，可以对已经存在的结构添加不同种类的新联系、新节点、新标签和新子图。
  • 图数据库是处理复杂的、结构可变的、密集关联的数据的最好的技术

图计算引擎

  • 偏重于全局查询
  • 通常为扫描和批处理大规模信息做过优化

图建模

  • 带标签的属性图模型
  • 图查询语言cypher
  • 建模是为了让不规则的领域的一些具体方面变成结构化的、可操从的空间
  • 图建模与其他建模技术的不同之处在于其逻辑模型和物理模型之间有更加密切的关系

带标签的属性图模型

  • 带标签的属性图是由节点、联系、属性和标签组成
  • 节点上包含属性。我们可以把节点想象成存储属性的文件，属性可以以任何键值对的形式存在，在Neo4j中，键一般是字符串，值则可以使java字符串和基元数据，或是这些数据类型的数组
  • 节点可以被打上一个或多个标签。标签把节点组织在一起，并表示它们在这个数据集中的角色
  • 联系连接节点，从而构成图。每条联系都有一个方向、一个名字、一个开始和一个结束节点
  • 联系也有属性，通过在联系上增加属性，可以给图算法提供元数据，也可以给联系增加额外的语义，还可以用于运行时的约束查询

映射概念

  • 映射是定义文档以及文档包含的字段如何被存储和索引的过程

    3、时间的格式

    4、用于控制动态添加字段的映射的自定义规则。

  • 每个索引包含一个映射类型

    1、Meta-fields

    2、Fields or properties

定义映射

字段类型

核心类型

  • text

    net_analyzed:分析过程被略过，整个字符串被当做单独的词条进行索引
    no:索引被忽略，即无法在该字段索引

  • 数值

  • 数组

  • 多字段

  • 控制文档路由到哪些分片

更新现有文档

文档的更新包括检索文档、处理文档、并重新索引文档，直至先前的文档被覆盖

  • es自动重试

  • 并发版本控制

删除文档

  • 删除单个文档或一组文档

    es只是标记为删除，只是不会出现在搜索结果中，稍后es通过异步的方式彻底地从索引中移除

    删除文档可能的问题是，删除文档后，再通过更新添加文档，为了防止这一问题的发生，通过index.gc_deletes设置删除文档的保留时间

  • 删除整个索引

    性能更好，主要工作就是移除和那个索引相关的文件

    文件只是被标记为已删除，在分段进行合并时，才被真正删除

    合并是指将多个lucene小分段组合为一个更大的分段的过程

  • 关闭索引

    关闭的索引不允许读取或者写入操作，数据也不会加载到内存，还是会保留到磁盘上，只需要打开索引即可恢复

  • 删除类型

    类型名称只是文档中的另一个字段。

    索引中的所有文档，无论它们属于哪个映射类型，都存放在同一分片中。

    es只能查询属于那个类型的文档，然后删除它们

    因此，删除类型通常要耗费更长的时间和更多的资源

分段与合并

分段就是建立索引的时候所创建的一块lucene索引（es成为分片）。

es概览

发表于 2020-07-10 更新于 2021-06-20 分类于 es 阅读次数：

简介

计算文档相关性得分的算法：TF-IDF(term frequency-inverse document frequency)词频-逆文档频率

  • 词频：所查找的单词在文档中出现的次数越多，得分越高
  • 逆文档词频：如果某个词频在所有文档中比较少见，那么该词的权重越高，得分也会越高

使用案例：

  • 不光能在完善的数据存储之上提供搜索能力，更能提供持久化数据的能力、统计能力
  • 版本1中不支持事务和复杂关系
  • 提供REST API服务，返回数据格式可以使JSON或YAML

主要特性：

  • es默认是面向文档的、可扩展的，没有固定的schema
  • 在多个字段中搜索多个关键字，一些是必须的，一些是可选的
  • 搜索可以同时包含查询、过滤器、聚集、聚集从匹配的文档中生成统计数据
  • 以文档的形式存储数据，文档可以是具有层次的，这种灵活性非常高，鼓励将所有属于一个逻辑实体的数据保存在同一个文档中。
  • 每个集群都有一个主节点，负责了解集群中的所有节点以及分片位置
  • 端口9300用于节点之间的通讯、java api
  • 端口9200用于http通讯
  • gateway负责将数据持久化到磁盘
  • es自动将数据划分为分片，做负载均衡，使得动态添加和移除服务器变动容易，分片可以复制，提高容错性

功能

逻辑设计

  • 文档是搜索和索引的基本单位，类似关系数据库中的一行
  • 类型用来分组文档，可以包含若干文档，类似表格包含若干行，类似表
  • 索引是更大的容器，一个或多个类型位于同一索引中，类似数据库

物理设计

  • es将每个索引划分为分片
  • 每份分片可以在集群间的不同服务器上迁移
  • 物理配置的设计方式决定了集群的性能、可扩展性和可用性

文档

  • 文档由索引名称 + 类型名称 + 文档ID 唯一确定
  • 文档是无模式的，不是所有文档都需要相同的字段
  • 文档是层次型的，即文档中可以包含文档
  • 文档拥有灵活的结构，不依赖于预先定义的模式
  • 文档中的字段是有字段类型的，字符串、整数等
  • 一个词条是文本中的一个单词，是搜索的基本单位

类型

  • 类型是映射类型，要和字段类型区分开来
  • 类型是文档的逻辑容器，类似于表格是行的容器。
  • 相同结构（模式）的文档最好是放入到相同的类型中
  • 映射是指每个类型中字段的定义，包含类型中所有文档的所有字段
  • 多个类型中出现同名字段时，同名字段应该有同样的类型，因为两个字段属于同一个lucene索引，若类型不同，es将很难辨别

既然es是无模式的，那么相同结构（模式）的文档要放到同一类型中吗？

  • 无模式是指文档不受模式的限制，文档不需要映射中定义的所有字段，当然也可以使用新字段。
  • 既然每个字段都需要在类型中定义映射，那么对于新字段，es会自动地将新字段加入映射
  • es在加入新字段时，将会猜测该字段的字段类型，可能会猜不对
  • 最佳的方式是索引数据之前，就定义好所需映射

索引

  • 索引是映射类型的容器
  • 是独立的大量文档的集合
  • 每个索引存储在磁盘上的同组文件中
  • 索引可以设置分片的数量

节点

  • 一个节点是一个es实例
  • 多个节点可以加入同一个集群
  • 默认情况下，可以连接集群的任一节点，并访问完整的数据集

分片

  • 分片是es处理的最小单元
  • 一份分片是一个lucene的索引：一个包含倒排索引的文档目录
  • 一个索引（es）是由多个lucene的索引组成
  • 主分片必须是在创建索引之前就确定
  • 过少的分片影响扩展性，过多的分片影响性能
  • 新节点加入到集群后，现有的分片将在所有节点上进行负载均衡
  • 对于每篇文档，分片是通过其ID字符串的散列决定的
  • 每份分片拥有相同的散列范围，接受新文档的机会均等

副本分片

  • 副本分片是主分片的完整副本
  • 副本分片用于搜索，或主分片丢失后成为新的主分片
  • 副本分片可以在任何时候进行数量的改变
  • 搜索时会通过轮询机制选中主分片和副本分片

操作

字段含义

  • _shard

    一共有多少分片，虽然默认有5个shard和3个replica，但是一个document只会存放在一个shard里面，而一个shard有多个replica，而我这只有个shard，没有replica，所有只有一个成功。

  • _version

    表示当前数据的版本号，数据发生变更时，递增

  • _seq_no

    严格递增的顺序号，每个文档一个，Shard级别严格递增，保证后写入的Doc的_seq_no大于先写入的Doc的_seq_no。任何类型的写操作，包括index、create、update和Delete，都会生成一个seq_no.这个是不可改的，而_version这个版本号是可以改的

  • _primary_term

    primary_term也和_seq_no一样是一个整数，每当Primary Shard发生重新分配时，比如重启，Primary选举等，_primary_term会递增1。_primary_term主要是用来恢复数据时处理当多个文档的_seq_no一样时的冲突，比如当一个shard宕机了，raplica需要用到最新的数据，就会根据_primary_term和_seq_no这两个值来拿到最新的document

  • acknowledged

    索引是否在集群中成功创建

  • shards_acknowledged

    超时前是否为索引中的每个分片启动了所需数量的分片副本。
