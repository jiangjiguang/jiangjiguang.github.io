设计模式

发表于 2021-01-04 更新于 2021-06-20 分类于 java 阅读次数：

创建型

  • 将创建和使用代码解耦

单例模式

  • 单例模式用来创建全局唯一的对象

工厂模式

  • 工厂模式用来创建不同但是相关类型的对象（继承同一父类或者接口的一组子类），由给定的参数来决定创建哪种类型的对象。

建造者模式

  • 把类的构造函数改为private

  • 类不提供任何set方法，这样创建出来的对象就是不可变对象

  • 先创建bulder类，通过set方法设置bulder类的变量，在bulder类真正创建对象之前，做集中的校验

    bulder是个静态内部类

  • 应用场景

    1、属性很多

    2、属性之间有依赖关系

    3、创建不可变对象

  • 与工厂模式的区别

    1、工厂模式是用来创建不同但是相关类型的对象，由给定的参数来决定创建哪种类型的对象

    2、建造者模式是用来创建一种类型的复杂对象，通过设置不同的可选参数，“定制化”地创建不同的对象。

原型模式

  • 如果对象的创建成本比较大，且同一类的不同对象的差别不大，这是可以利用复制的方式创建对象

  • 实现深拷贝

    1、递归拷贝对象

    2、先序列化，再反序列化

结构型

  • 将不同功能代码解耦

代理模式

  • 在不改变原始类（或叫被代理类）代码的情况下，通过引入代理类来给原始类附加功能

  • 如果没有办法修改原始类，那么代理类就继承原始类

  • 应用场景

    代理模式常用在业务系统中开发一些非功能性需求，比如：监控、统计、鉴权、限流、事务、幂等、日志

  • 动态代理原理解析

    1、动态代理：不事先为每个原始类编写代理类，而是在运行时，动态地创建原始类对应的代理类，用代理类替换掉原始类

    2、动态代理底层依赖的就是反射

桥接模式

  • 将抽象（类库）和实现解耦（类库），让它们可以独立变化。
  • 类似“组合优于继承”设计原则

装饰器模式

  • 装饰器模式不仅仅是“用组合替代继承”

    1、装饰器类和原始类继承同样的父类，这样可以对原始类“嵌套”多个装饰器类

    2、装饰器类是对功能的增强，这也是装饰器模式应用场景的一个重要特点

  • “组合结构”代码结构

    1、代理模式中，代理类附加的是跟原始类无关的功能

    2、装饰器模式中，装饰器类附加的是跟原始类相关的增强功能

  • 典型应用场景

    1、装饰器模式主要解决继承关系过于复杂的问题，通过组合来替代继承。

    2、它主要的作用是给原始类添加增强功能。这也是判断是否该用装饰器模式的一个重要的依据

    Java IO类

适配器模式

  • 用来做适配的，它将不兼容的接口转换为可兼容的接口

  • 应用场景

    1、封装有缺陷的接口设计

    2、统一多个类的接口设计

    3、替换依赖的外部系统

    4、兼容老版本接口

    5、适配不同格式的数据

  • 代理、桥接、装饰器、适配器 4 种设计模式的区别

    1、代理模式在不改变原始类接口的条件下，为原始类定义一个代理类，主要目的是控制访问，而非加强功能，这是它跟装饰器模式最大的不同。

    2、桥接模式的目的是将接口部分和实现部分分离，从而让它们可以较为容易、也相对独立地加以改变。

    3、装饰者模式在不改变原始类接口的情况下，对原始类功能进行增强，并且支持多个装饰器的嵌套使用。

    4、适配器模式是一种事后的补救策略。适配器提供跟原始类不同的接口，而代理模式、装饰器模式提供的都是跟原始类相同的接口。

门面模式

  • 可以用来封装系统的底层实现，隐藏系统的复杂性，提供一组更加简单易用、更高层的接口
  • 解决性能问题，将多个接口调用替换为一个门面接口调用
  • 尽量保持接口的可复用性，但针对特殊情况，允许提供冗余的门面接口，来提供更易用的接口

组合模式

  • 将一组对象组织（Compose）成树形结构，以表示一种“部分 - 整体”的层次结构

享元模式

  • 所谓“享元”，顾名思义就是被共享的单元
  • 享元模式的意图是复用对象，节省内存，前提是享元对象是不可变对象。
  • 它的代码实现非常简单，主要是通过工厂模式，在工厂类中，通过一个 Map 来缓存已经创建过的享元对象，来达到复用的目的。
  • 在Integer和String都有使用

行为型

  • 将不同的行为代码解耦

观察者模式（发布订阅模式）

  • 将观察者和被观察者代码解耦

  • 在对象之间定义一个一对多的依赖，当一个对象状态改变的时候，所有依赖的对象都会自动收到通知。

  • 实现方式

    1、同步阻塞

    2、异步非阻塞

模板模式

  • 模板模式主要是用来解决复用和扩展两个问题
  • 模板方法模式在一个方法中定义一个算法骨架，并将某些步骤推迟到子类中实现。模板方法模式可以让子类在不改变算法整体结构的情况下，重新定义算法中的某些步骤。

策略模式

  • 应用场景

    1、策略模式用来解决根据运行时状态从一组策略中选择不同策略的问题。工厂模式侧重封装对象的创建过程

    2、在策略模式中，不同的策略具有相同的目的、不同的实现、互相之间可以替换。

    ​ 在命令模式中，不同的命令具有不同的目的，对应不同的处理逻辑，并且互相之间不可替换

职责链模式

  • 将请求的发送和接收解耦，让多个接收对象都有机会处理这个请求。将这些接收对象串成一条链，并沿着这条链传递这个请求，直到链上的某个接收对象能够处理它为止。

状态模式

  • 状态模式一般用来实现状态机

  • 有限状态机：缩写为 FSM，简称为状态机

    1、状态机由3个部分组成：状态（State）、事件（Event）、动作（Action）

    2、事件也称为转移条件（Transition Condition）。事件触发状态的转移及动作的执行。

  • 状态机实现方式

    1、分支逻辑法

    2、查表法

    3、状态模式：通过将事件触发的状态转移和动作执行，拆分到不同的状态类中，来避免分支判断逻辑。

迭代器模式

  • 迭代器是用来遍历容器的，所以，一个完整的迭代器模式一般会涉及容器和容器迭代器两部分内容

访问者模式

  • 允许一个或者多个操作应用到一组对象上，解耦操作和对象本身

  • Single Dispatch

    执行哪个对象的方法，根据对象的运行时类型来决定；

    执行对象的哪个方法，根据方法参数的编译时类型来决定。

    之所以称为“Single”，是因为执行哪个对象的哪个方法，只跟“对象”的运行时类型有关

  • Double Dispatch

    执行哪个对象的方法，根据对象的运行时类型来决定；

    执行对象的哪个方法，根据方法参数的运行时类型来决定

    之所以称为“Double”，是因为执行哪个对象的哪个方法，跟“对象”和“方法参数”两者的运行时类型有关。

    不需要访问者模式

  • Java 语言只支持 Single Dispatch

备忘录模式

  • 在不违背封装原则的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态，以便之后恢复对象为先前的状态。

命令模式

  • 命令模式将请求（命令）封装为一个对象，这样可以使用不同的请求参数化其他对象（将不同请求依赖注入到其他对象），并且能够支持请求（命令）的排队执行、记录日志、撤销等（附加控制）功能。

  • 命令模式用的最核心的实现手段，是将函数封装成对象

  • 应用场景

    用来控制命令的执行，比如，异步、延迟、排队执行命令、撤销重做命令、存储命令、给命令记录日志等

解释器模式

  • 解释器模式为某个语言定义它的语法（或者叫文法）表示，并定义一个解释器用来处理这个语法。
  • 代码实现的核心思想，就是将语法解析的工作拆分到各个小类中，以此来避免大而全的解析类

中介模式

  • 中介模式定义了一个单独的（中介）对象，来封装一组对象之间的交互。将这组对象之间的交互委派给与中介对象交互，来避免对象之间的直接交互。

Iceberg基本知识

发表于 2020-12-31 更新于 2021-06-20 分类于 iceberg 阅读次数：

简介

  • Iceberg是一种可伸缩的表存储格式，用于大型数据集分析

  • Table 演变，

    1、可以演变表的schema

    2、可以演变表的分区布局Schema演变

Table Evolution

  • Iceberg内在支持表演变

  • Schema evolution

    Add：向表或嵌套结构中添加新列

    Drop：从表或嵌套结构中移除已存在的列

    Rename：重命名嵌套结构中的现有列或字段

    Update：widen the type of a column, struct field, map key, map value, or list element

    Reorder：更改嵌套结构中列或字段的顺序

Partitioning

  • 在hive中分区字段是被显示声明并且作为一列的，缺点

    1、字段之间不存在关联

    2、hive不能验证分区字段的值的正确性

    3、在查询期间，分区字段是不能被改变的

  • Iceberg隐藏了分区字段

    1、iceberg的分区字段由其它字段转换而来

    2、iceberg并不保存分区字段

    3、iceberg的查询不再依赖物理分区布局

Time travel

  • 使用相同的table snapshot复用查询，并且容易检查改变

Version rollback

  • 允许用户通过将表重置为良好状态来快速纠正问题

Scan planning

  • 是指在表中查找查询所需文件的过程

  • planning适合单节点

  • Metadata filtering：Iceberg使用两层metadata来追踪快照中的文件

    1、Manifest files：存储数据文件列表，以及每个数据文件的分区数据和列级统计数据

    2、manifest list：存储manifests文件的快照列表，以及每个分区字段的值的范围

  • scan planning过程

    1、在manifest list中通过分区值的范围过滤manifests文件

    2、读取manifest，获取数据文件

  • Data filtering

    1、planning期间，查询谓词将自动转换为分区数据上的谓词，并首先应用于过滤数据文件。

Hive存在的问题

  • 表状态在两个地方存储

    1、在hive metastore中存储分区

    2、在文件系统中存储文件

  • Bucketing is defined by Hive’s (Java) hash implementation

  • Non-ACID layout’s， only atomic operation is add partition

  • Requires atomic move of objects in file system

  • Still requires directory listing to plan jobs

Pushgateway学习笔记

发表于 2020-12-14 更新于 2021-06-20 分类于 prometheus 阅读次数：

基本概念

  • Pushgateway是Prometheus生态中的一个重要工具

  • 客户端调用Pushgateway的接口向Pushgateway推送指标，Prometheus定时拉取指标

  • Pushgateway适用于Prometheus无法直接拉取各个target的指标，汇总数据

  • 缺点

    1、Pushgateway是单点的

    2、当数据量特别大时，Pushgateway会占用大量内存，导致Prometheus拉取超时

    3、Pushgateway不会自动删除过期数据，需要外部编写脚本删除

使用实践

  • 推送到Pushgateway的指标按组（group）进行管理，并由任意数量的group key labels标识，其中job必须是第一个label

    1、Prometheus server 会附加 job和instance label到每个指标数据上

    2、job label可以来自指标数据（必须设置honor_labels: true），也可以从配置中取

    3、instance 自动是target的host和port

  • 不管group是否相同，相同名称的指标必须是相同的类型

目的

​ 在建设实时计算平台过程中，经常需要根据自身需求修改flink源码，而这些源码是不需要共享给社区的。那么如果管理源码，多人协作开发就是需要考虑的问题。以下是要实现的目的：

  • flink代码仓库建在公司的git仓库中
  • 实现多人协作开发，开启分支保护，新增代码必须经过review才能合并到生产分支中
  • 可以从官方 github apache/flink中同步代码
  • 可以把代码push到github 的个人私有仓库中

  • 尽量每个功能一次commit，即使为了修改bug进行了多次commit，也要在push的时候合并多次commit为一次

  • 每次合并基础分支时使用rebase，不要使用merge
