CPU探究

发表于 2020-09-22 分类于 linux 阅读次数：

原理介绍

实用工具

async-profiler

  • 这是一个用于Java的低开销的采样探测器，没有Safepoint bias问题

  • 适用于HotSpot JVM，收集stack traces and to track memory allocations

  • 可以跟踪以下事件

    1、CPU cycles(CPU周期)

    2、硬件和软件性能计数器，例如高速缓存未命中，分支未命中，页面错误，上下文切换等。

    3、Java堆分配

    4、Contented lock attempts，包括Java对象监视器和ReentrantLocks

CPU profiling

  • 在这种模式下，profiler收集stack trace samples

    包含了 Java methods, native calls, JVM code and kernel functions.

  • 通常的方法是接收perf_events生成的调用堆栈，并将其与AsyncGetCallTrace生成的调用堆栈进行匹配

  • 与直接将perf_events与将地址转换为Java方法名称的Java代理一起使用相比，此方法具有以下优点：

    1、不需要-XX:+PreserveFramePointer 参数，可以用在JDK 8u60 之前的版本

    2、不需要-XX:+PreserveFramePointer 参数，提高了性能

    3、不需要生成将Java代码地址映射到方法名称的映射文件。

    4、Works with interpreter frames.

    5、不需要写出perf.data文件以在用户空间脚本中进行进一步处理。

ALLOCATION profiling

  • 除了可以进行CPU消耗检测，还可以收集分配了最大堆内存的调用站点

    async-profiler不使用侵入性技术，例如字节码检测工具或昂贵的DTrace探针，这些技术会对性能产生重大影响。它也不会影响转义分析或防止JIT优化（如分配消除）。 仅测量实际堆分配。

  • profiler具有TLAB驱动的采样功能。 它依赖于HotSpot特定的回调来接收两种通知：

    1、在新创建的TLAB中分配对象时（火焰图中的浅绿色帧）；

    2、当在TLAB之外的慢路径上分配对象时（棕色框架）。

    这意味着不计算每个分配，而仅计算每N kB的分配，其中N是TLAB的平均大小。

    这使得堆采样成本低并且适合于生产。 另一方面，所收集的数据可能不完整，尽管在实践中通常会反映出最主要的分配来源。

  • 可以使用-i选项来调整采样间隔。

    例如，-i 500k将平均分配500 KB的分配空间后进行一次采样。 但是，小于TLAB大小的间隔不会生效。

    与使用类似方法的Java Mission Control不同，async-profiler不需要Java Flight Recorder或任何其他JDK商业功能。 它完全基于开源技术，并且可以与OpenJDK一起使用。

  • 支持的最低JDK版本是7u40，其中会出现TLAB回调。

Installing Debug Symbols

  • 分配分析器需要HotSpot调试符号。 Oracle JDK已经将它们嵌入libjvm.so中，但是在OpenJDK构建中，它们通常以单独的软件包提供。

Wall-clock profiling

  • -e wall选项告诉async-profiler在每个给定的时间段内平均采样所有线程，而不管线程状态如何：正在运行，正在睡眠或已阻塞。 例如，在分析应用程序启动时间时这可能会有所帮助。

  • Wall-clock Profiler在每个线程模式下最有用：-t

    Example: ./profiler.sh -e wall -t -i 5ms -f result.svg 8983

Java method profiling

  • -e ClassName.methodName选项用于检测给定的Java方法，以便使用堆栈跟踪记录该方法的所有调用。
  • Only non-native Java methods are supported. To profile a native method, use hardware breakpoint event instead, e.g. -e Java_java_lang_Throwable_fillInStackTrace

Building

Basic Usage

  • start

    starts profiling in semi-automatic mode, i.e. profiler will run until stop command is explicitly called.

  • resume

    starts or resumes earlier profiling session that has been stopped. All the collected data remains valid. The profiling options are not preserved between sessions, and should be specified again.

  • stop

    stops profiling and prints the report

  • check

    check if the specified profiling event is available

  • status

    prints profiling status: whether profiler is active and for how long

  • list

    show the list of available profiling events. This option still requires PID, since supported events may differ depending on JVM version

  • -d N

    the profiling duration, in seconds. If no start, resume, stop or status option is given, the profiler will run for the specified period of time and then automatically stop

    ./profiler.sh -d 30 8983

  • -e event

    the profiling event: cpu, alloc, lock, cache-misses etc. Use list to see the complete list of available events

Transformation

  • 一共有11种Transformation

    1、PartitionTransformation：不是PhysicalTransformation，仅仅作用是上游operation如何连接下游operations

    2、SourceTransformation：是PhysicalTransformation，不会转变任何数据，但是是拓扑的root

    3、OneInputTransformation：是PhysicalTransformation

    4、TwoInputTransformation：是PhysicalTransformation

    5、SinkTransformation：是PhysicalTransformation

    6、UnionTransformation：不是PhysicalTransformation，多个Transformation的联合，仅仅作用是上游operation如何连接下游operations

    7、SplitTransformation：不是PhysicalTransformation，是将一个DataStream转化为多个DataStream，仅仅作用是上游operation如何连接下游operations

    8、SelectTransformation：不是PhysicalTransformation，选择上游的某种元素，然后转换成另一个DataStream

    9、FeedbackTransformation

    10、CoFeedbackTransformation

    11、SideOutputTransformation：不是PhysicalTransformation

  • 重要属性

    1、每个Transformation都有个managedMemoryWeight的属性，也会设置到StreamGraph

    2、每个Transformation都有个input属性，input也是Transformation，是该Transformation的上游

    3、所有的Transformation是按照sink的在外层，上游是其input的规则链在一起

  • Transformation就是创建DataStream的一个operation

  • Transformation分为：PhysicalTransformation和非PhysicalTransformation

  • addOperator方法的参数就是Transformation

    创建operators必须调用此方法

内存探究

发表于 2020-09-08 更新于 2020-09-10 分类于 linux 阅读次数：

基本概念

主存

也称为物理内存，是计算机的数据存储区域。通常是动态随机访问内存（DRAM）

虚拟内存

  • 是一个抽象概念
  • 大多数虚拟内存是以页的方式实现的
  • 它向每个进程和内核提供巨大的、线性的并且私有的地址空间
  • 它简化了软件开发，把物理内存的分配交给操作系统管理
  • 支持多任务，因为虚拟地址空间被设计成分离的，并且可以超额订购
  • 进程的地址空间由虚拟内存子系统映射到主内存和物理交换设备，内核按需在它们之间移动内存页，称为交换

常驻内存

当前处于主存中的内存

匿名内存

无文件系统位置或路径名的内存，包括进程地址空间的工作数据，称为堆。

地址空间

内存上下文，每个进程和内核都有对应的虚拟地址空间

段

标记为特殊用途的一块内存区域，例如用来存储可执行或者可写的页

页

操作系统和CPU使用的内存单位。它一直以来是4KB或者8KB。允许多种页大小以支持更大的页面尺寸

缺页

无效的内存访问，使用按需虚拟内存时，这是正常事件

换页

  • 换页是将页换入和调出主存，分为称为页面换入和页面换出
  • 由于页的尺寸较小，换页是精确管理和释放主存的手段
  • 有两种类型的换页：文件系统换页和匿名换页

文件系统换页

  • 文件系统换页由读写位于内存中的映射文件页引发，对于使用文件内存映射（mmap()）的应用程序和使用了页缓存的文件系统，这是正常行为。是好的换页。

匿名换页

  • 匿名换页牵涉进程的私有数据：进程堆和栈
  • 被称为匿名是由于它在操作系统中缺乏有名字的地址
  • 匿名页面换出要求迁移数据到物理交换设备或者交换文件，linux称为swapping
  • 匿名换页拖累性能，是坏的换页

按需换页

  • 是指支持按需换页的操作系统将虚拟内存按需映射到物理内存

  • 把CPU创建映射的开销推迟到实际需要或访问时，而不是在初次分配这部分内存时

  • 当访问一个尚未从虚拟映射到物理内存的页时，会发生缺页

  • 如果这个映射可以由内存中其它的页满足，称为轻微缺页

  • 需要访问存储设备的缺页，称为严重缺页

  • 虚拟内存和按需换页的结果是任何虚拟内存也可能处于如下的一个状态：

    A. 未分配

    B. 也分配，未映射

    C. 已分配，已映射到主存（RAM）

    D. 已分配，已映射到物理交换空间（磁盘）

  • 常驻集合大小（RSS）：已分配的主存页（C）大小

  • 虚拟内存大小：所有已分配的区域（B+C+D）

过度提交

  • 是指允许分配超过物理内存和交换设置的总和
  • 依赖于按需换页以及应用程序通常不会使用分配给他们的大部分内存
  • linux可以用可调参数配置过度提交
  • 过度提交的后果取决于内核如何管理内存压力

性能调优

特别说明

nice值

​ 进程特性nice值允许进程间接的影响内核调度算法。所谓nice就是指对其他进程的谦让程度，显然比0越小就越不谦让，比0越大就越谦让。顾名思义，越谦让的君子在CPU调度过程中占用CPU的时间会越少，反之不谦让的进程相比较可能占用更多的CPU时间。因此字符’<’表示此进程可能在调度过程中获得优势。

缺页

​ 操作系统管理内存的基本单元是4KB大小的页框，当进程访问尚未有物理内存建立页表映射关系的虚拟内存地址值时，会产生一次缺页异常。在缺页异常处理过程中会为虚拟内存页分配一个物理内存页并建立映射。所以每一次缺页异常就会分配4096（4kb）字节的物理内存，这样rss和minflt的比值当然就是4了。如果分配之后又有释放，后面再次分配，会使这个比值逐步小于4。如果这个比值过于小，那我们就有充足理由怀疑用户程序代码在内存管理上存在重大问题

引用

Java中提供了4个界别的引用：强引用、软引用、弱引用和虚引用

##强引用（Strong Reference）

  • 在一个线程内，可以直接使用的对象

软引用（Soft Reference）

  • 虚拟机会尽量让软引用的对象存活时间长一些
  • 发生OOM之前会清理所有的软引用对象
  • 虚拟机在GC时会通过算法检查是否回收软引用对象
  • 虚拟机在回收时会将引用存放到一个引用队列

弱引用（Weak Reference）

  • 虚拟机在GC时总是回收弱引用对象
  • 复杂关系的弱引用对象常常需要几次GC才能完成回收

虚引用（Phantom Reference）

  • 虚引用对象get方法返回总是null，因此无法获取被引用的对象
